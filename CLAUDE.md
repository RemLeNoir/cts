# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo actually is

A **library of semantic contracts for working with AI**, in Spanish. The headline deliverable is a markdown table per domain that pairs "lo que se suele decir" (vague, lazy prompt) with "lo que se debería decir" (operative, verifiable prompt). The canonical source lives at [biblioteca/biblioteca.md](biblioteca/biblioteca.md); a frontend serves it at [index.html](index.html) via [biblioteca/biblioteca.json](biblioteca/biblioteca.json), generated by [scripts/build_biblioteca.py](scripts/build_biblioteca.py).

There is also a smaller, optional **formal layer** carried over from the original spec ([.docs/cts.md](.docs/cts.md)): a JSON Schema, three templates, six example contracts, and a hardcoded simulator. Treat it as legacy / power-user surface; the day-to-day product is the biblioteca.

## Three eras of the spec — read this before editing

The repo has evolved through three drafts in [.docs/](.docs/):

1. [.docs/cts.md](.docs/cts.md) — the original spec. Described a full repo of formal contracts (schema, templates, examples, validators). **Materialized in the first commit on main**.
2. [.docs/biblioteca.md](.docs/biblioteca.md) and [.docs/biblioteca2.md](.docs/biblioteca2.md) — interim drafts as the user pivoted toward the prompt-library model.
3. [.docs/biblioteca3.md](.docs/biblioteca3.md) — current canonical source for the library, copied into [biblioteca/biblioteca.md](biblioteca/biblioteca.md) as the working file. **The biblioteca is now the main product**.

If the user adds a `biblioteca4.md` (etc.), treat it as the new canonical source and propagate it into [biblioteca/biblioteca.md](biblioteca/biblioteca.md). Always run `python scripts/build_biblioteca.py` afterward.

A failed branch attempt to deepen the formal layer (schema v2, composition engine, base contracts) was discarded because the user's actual product was the library, not a formalism on top of it. Don't reintroduce schema v2 / `compose.py` / `templates/bases/` without an explicit ask.

## Architecture: how the biblioteca pipeline fits together

```
biblioteca/biblioteca.md     ← canonical (humans edit here)
       │
       ▼  python scripts/build_biblioteca.py
biblioteca/biblioteca.json   ← generated, committed
       │
       ▼  fetch()
index.html                   ← static frontend (no build, no deps)
       │
       ▼  GitHub Pages
https://remlenoir.github.io/cts/
```

[scripts/build_biblioteca.py](scripts/build_biblioteca.py) parses the markdown by walking `## N. Title` headings, extracting tables, and tagging each section as `transformaciones` (default), `meta` (section 0, "Las seis cláusulas"), `antipatrones` (when the second column reads "Por qué no tiene versión válida"), or `regla` (sections without a table — the closing rule).

The build is **idempotent up to the `generado` timestamp**. The test [tests/test_biblioteca.py](tests/test_biblioteca.py) re-runs the build in memory and compares to the committed JSON, ignoring `generado`. If you edit the markdown without running the build, that test fails with a clear message.

The frontend ([index.html](index.html)) is one self-contained file: no build step, no npm, no framework. It fetches the JSON, renders cards with domain chips for filtering, supports live search (`/` focuses the input, `Esc` clears), and click-to-copy on the "lo que se debería decir" column. The legacy `panel-traduccion` files were deleted because the biblioteca subsumes them.

### Coupling rules to keep in mind

- **Editing the biblioteca markdown without rebuilding** → `test_json_commiteado_coincide_con_fuente` fails. Always run `python scripts/build_biblioteca.py` after edits.
- **Adding new section types** beyond `transformaciones / meta / antipatrones / regla` requires updating both the parser (`parse_section` in `build_biblioteca.py`) and the frontend (`renderSection` in `index.html`).
- **Section IDs are derived from `<numero>-<slugified-title>`**. If you renumber sections, all IDs shift, which breaks deep links into the live site. Avoid renumbering once published.
- **Tests are sensitive to column counts** — every row must have exactly 2 cells; every section header must have exactly 2 columns. The parser is tolerant (skips malformed rows) but the test asserts strict shape.

## The formal layer (legacy, but functional)

These pieces remain from the original spec and still work; the tests pass against them. Don't expand them, but don't break them either:

- [schemas/contrato.schema.json](schemas/contrato.schema.json) — JSON Schema v1 (loose).
- [templates/contrato-semantico.{md,json,yaml}](templates/) — fill-in-the-blank skeleton.
- [examples/<dominio>/](examples/) — 6 domain examples with `README.md` + `test_case.json`.
- [scripts/validate_contracts.py](scripts/validate_contracts.py) — walks templates+examples, validates against schema.
- [scripts/simulate_model.py](scripts/simulate_model.py) — hardcoded `DOMAIN_SECTIONS` dict per domain. **Token collision gotcha**: in the legal example, an earlier draft had "riesgos interpretativos" in the Supuesto content, which collided with the later `riesgo` priority. Fixed to "dudas interpretativas"; preserve that.
- [scripts/run_example_tests.py](scripts/run_example_tests.py) — feeds test cases through the simulator, asserts token presence / order / count.

## Commands

```bash
# Dev deps (Python 3.11+)
python -m pip install -U jsonschema pyyaml pytest yamllint

# Rebuild the biblioteca JSON after editing biblioteca/biblioteca.md
python scripts/build_biblioteca.py

# Full test suite (includes biblioteca, schema, examples, presence)
pytest -q

# Single test file or test
pytest tests/test_biblioteca.py
pytest tests/test_biblioteca.py::test_json_commiteado_coincide_con_fuente

# Validate formal contracts against the schema
python scripts/validate_contracts.py

# Example-driven behavioural tests for the formal layer
python scripts/run_example_tests.py

# YAML lint (CI uses yamllint .)
yamllint .

# Serve the frontend locally for hand-testing
python -m http.server   # then open http://localhost:8000
```

### Windows console encoding

The Python scripts print a UTF-8 check mark (`✔`). On Windows shells whose default code page is cp1252, set `PYTHONIOENCODING=utf-8` before running anything that prints to stdout, or stdout will raise `UnicodeEncodeError`. CI on Linux is unaffected.

## Conventions worth preserving

- Folder/file names: lowercase, hyphen-separated. Section IDs in the biblioteca follow the same convention via `slugify()`.
- The biblioteca is **Spanish-only** content. Match register: direct, concrete, no padded jargon. If a word like "claro" or "profesional" carries weight, define it.
- The biblioteca's closing **"Regla transversal"** ("Todo encargo se cierra con validación humana. La IA no firma, no decide y no es la fuente de verdad…") is the project's North Star. Any new section or row must respect it.
- Section 18 / 28 / wherever-it-lands of the biblioteca is **"Antipatrones sin versión válida"**: instructions whose objective is itself invalid. These are catalogued, not "improved". Don't try to write a "should-say" column for entries that belong here.
- Don't add MVP/phased/roadmap concessions to the biblioteca content. Section "Specs y diseño de producto" explicitly rejects MVP framing; honor that throughout.
